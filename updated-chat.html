Economics

Allocation Model

Three-stage distribution confirmed across six team calls. All variables dynamic, never hard-coded.

Core parameters
Target raise: $500,000 (scalable per project)
Token supply: ~460M circulating of 1B total
Expected staked: ~60M tokens
Median lock: 180 days
Investment currency: USDT/USDC (not BLOCK)
Base allocation: ~$50 per user at $500K raise
Min allocation floor: >$10 (prevents dust)
Participation rate: 50–75% expected
Three-stage allocation
Stage 1 — First 24h

Guaranteed allocation

Qualified users (min stake, top leaderboard) get ranked allocation based on points. Maps to Phase 1.

Stage 2 — Next 24h

FCFS (stakers)

~20% of total. All stakers meeting minimum. First come, first serve on remaining pool. Maps to Phase 2.

Stage 3 — After

Open FCFS

Any leftover opens to all KYC-verified users regardless of holdings. Rare in hot markets.

Fee structure
Platform fee

7.5% of stablecoin raised

Paid by project. Industry-standard, set lower initially to build adoption. Room to increase as brand strengthens.

Token allocation

5% of project tokens

Negotiated per project. Covers incubation: marketing, liquidity provision, KOL connections. Contributes to liquidity pools.

Key principle: BLOCK = access/staking token. Stablecoins = investment currency. No buy/sell tax on BLOCK trading. Early unstaking penalty only.

Rationale

Key Decisions

The thought process behind what was decided and why — sourced from six team calls (Jan 26 – Mar 24).

D-01 · Confirmed

Remove multi-tier system, replace with qualified / not-qualified

Kyle C pushed for simplification. Multiple tiers created allocation complexity and oversubscription edge cases. A single qualification threshold with leaderboard ranking within the qualified group achieves the same result with less confusion.
Jan 26, confirmed Mar 19
D-02 · Confirmed

Top-500 leaderboard for guaranteed allocation, remainder enters FCFS

Hybrid model: top N users by points get guaranteed slots, everyone else competes in FCFS (~20%). Balances rewarding committed stakers with accessibility. Cutoff is configurable per launch.
Mar 19, refined from Jan 26 three-stage model
D-04 · Confirmed

VIP deposits committed 3 days before public sale (100%)

Kyle C insisted on 100% commitment (not David's 20%). Knowing exact VIP participation enables clean reallocation of unused funds. Prevents over-allocation and early sell-offs.
Jan 29
D-06 · Confirmed

Positive multipliers only — no penalties for selling

Penalizing users for selling resembles "meme token" mechanics, damages public perception, and adds complexity. Wallet analysis identifies and rewards positive behaviors. Only early unstaking incurs a penalty.
Mar 19, reinforced by Jan 29 no-tax decision
D-07 · Confirmed

Wallet quality, social, sector-fit multipliers deferred to post-V1

Kyle C: "Don't show features until fully functional." The allocation model works with zero soft multipliers. Hide these from UI until ready.
Most recent call, echoes Mar 24
D-08 · Confirmed

All model variables must be dynamic, never hard-coded

Prior incidents where hard-coded values caused system failures. UI must read from configurable parameters. Matheus and Sagar coordinate which inputs remain dynamic.
Mar 24
D-10 · Confirmed

Projects raise in stablecoins (USDT/USDC), not BLOCK

Kyle C: avoids treasury volatility risk. BLOCK = staking to qualify, not investment currency. Simplifies treasury management. Clean separation of access token vs. investment currency.
Jan 26
D-12 · Confirmed

Two-phase model. Phase 2 can repeat. No Phase 3.

Phase 1 = ranked allocations. Phase 2 = FCFS. Phase 2 repeats recursively if needed. Simplifies rollout while maintaining flexibility.
Most recent call
D-20 · Confirmed

Solana integration is top priority for V1

Unification with EVM launchpad is highest chain priority. Solana code changes are mostly removals, reducing audit complexity. Halborn engaged for audits.
Most recent call, Mar 12
D-22 · Confirmed

Target 1–2 high-quality projects per month

Quality over quantity. Platform builds reputation through strong returns, not volume. This means manual KYB is acceptable — volume is low and intentional.
Jan 29
D-25 · Open

Lottery system: undefined, code incomplete

Targets smaller holders without guaranteed allocations. Code partially exists but isn't connected at app level. Needs math justification. Not a V1 blocker but needs roadmap clarity.
Mar 12
D-28 · Deferred

Social XP / social mining as separate engagement system

On-chain wallet analysis and social engagement are distinct streams. Social mining is tied to user accounts (not wallets), circumventing multi-wallet gaming. Post-V1.
Mar 19, deferred in most recent call
