Access System

Vault Logic

A cleaner summary of how vault positions translate into access.

1

Vault Amount

User allocates BLOCK across one or more vault positions.

2

Vault Duration

Each vault duration carries its own fixed multiplier, for example 90d, 180d, or 365d.

3

Vault Points

Each vault generates points based on BLOCK amount multiplied by that vault’s duration multiplier.

4

Total Points

Total points equal the sum of all active vault positions across the account.

5

Rank

Total points contribute to leaderboard rank, which affects access priority and eligibility strength.

6

Pool Access

Rank, points, KYC, pool phase, chain, and wallet readiness together determine participation access.

Chain Support

Solana Layer

A separate, lightweight chain layer without breaking the main product logic.

Wallet model

Main wallet is always EVM and acts as the identity and vault reference wallet. If a pool is Solana-based, the user adds a Solana wallet as an optional participation wallet for that chain only.

Main wallet = EVM reference Optional Solana wallet Chain-specific participation
Pool-level behavior

Every pool card should expose a visible chain tag and a required wallet state. The CTA must adapt depending on whether the user is invested, not invested, or missing the required chain wallet.

Sold Out • Invested Sold Out • Not Invested Requires Solana wallet
Chain tag

Use a clear tag like EVM or Solana on all cards and detail pages so the required wallet path is obvious before the user clicks in.

If invested

Show Sold Out • Invested, then route to token claim or post-invest detail using the wallet context tied to that pool’s chain.

If not invested

Show Sold Out • Not Invested and replace invest actions with Buy on DEX. The destination should be chain-aware.

If wallet missing

For a Solana pool without a connected Solana wallet, the CTA should say Add Solana Wallet. Do not fail silently.

Execution

Priority

Ordered by structural impact, not visual polish.

Phase 1 — Fix Core Logic

Replace staking and tier language with vault logic

Update homepage, profile, and supporting product copy to reflect vault positions, points, and access status.

Remove tier card from profile

Replace it with an Access Status module based on vault totals, points, effective multiplier, rank, and access status.

Implement manual-claim pool states

Support notify, active invest states, sold-out branches, manual token claim, and manual refund claim.

Fix CTA guidance everywhere

Every blocked action should explain the exact blocker, KYC, points, phase, or wallet readiness.

Phase 2 — Add Missing Product Layers

Build dashboard page

Create the authenticated overview for access status, open pools, wallet readiness, and recent activity.

Add chain-aware wallet readiness

Make the EVM main-wallet model explicit and add optional Solana wallet flows where required.

Restructure navigation

Merge Pools, rename Vaults, and move founder creation into Portfolio.

Phase 3 — Polish

Refine claim messaging

Make Token Claims and investment-end states more contextual with better labels, timing, and next-action wording.

Decisions Needed

Open Questions

The few remaining decisions needed before the system is fully locked.

Q1

What is the final homepage framing for vault logic?

Should the front-end message emphasize vault rewards, points-based access, or a broader launch access system?

Q2

Can we confirm that effective multiplier should be shown as total points divided by total BLOCK vaulted?

This seems like the strongest summary metric for users with multiple vault positions, but it should be formally locked.

Q3

How visible should per-vault breakdowns be outside the Vaults page?

Dashboard should likely stay summary-level, but Profile or Vaults may expose each vault row and multiplier in detail.

Q4

Should dashboard replace homepage for authenticated users?

This will define the default post-login landing behavior and may affect the whole navigation model.

Q5

How should the chain-aware DEX CTA be labeled at launch?

Generic Buy on DEX is clean, but exact routing labels for Solana and EVM can be finalized later.
